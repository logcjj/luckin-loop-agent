3. 优化 UI 信息架构：在首屏展示“业务闭环 + Agent 信心 + 推荐理由”，让对话、方案、画像、执行、复购都更充实；默认体验应是产品工作台，不是纯执行舱。
4. 引入商品视觉资产与更真实的商品信息展示：商品图片、风味/营养/价格/来源说明、可替换的真实数据 adapter。
5. 加入地理位置体验：浏览器定位授权、附近门店 mock/真实接入边界、拒绝授权 fallback。
6. 重构自动执行模块：构建“瑞幸手机大窗 + GUI-Agent 执行小窗”的模拟点单流程，显示观察、动作、目标、坐标/元素、等待、返回、完成验证与支付前停止；该模块只在用户点击执行或切换自动执行时出现。
7. 优化美术风格：更接近真实瑞幸消费场景与高质量 demo shell，而不是简单蓝白卡片；保留克制动画和明确视觉叙事。
8. 做移动端适配：rail/canvas 在窄屏重排为可横向选择或堆叠布局，保持按钮和文字不溢出；Agent 对话要提供手机端 App-like 视图，不只是网页缩小。
9. 完善文档：说明架构、数据来源边界、真实系统接入方式、演示脚本和评估指标。
10. 验证：TypeScript build、页面截图/响应式检查、手动功能路径。

## 风险

- 直接抓取或声称使用真实瑞幸会员信息会有隐私和合规风险。
- 公开菜单、价格、优惠会随时间变化，必须避免写死为“官方实时”。
- 商品图片有版权和授权风险，不能随意复制第三方素材到项目中；若引用远程公开图片，要注明来源与替代方案。
- 浏览器地理位置涉及用户敏感权限，必须提供拒绝授权 fallback，并避免持久化精确坐标。
- 自动点单模拟不能暗示真的操作用户手机或真实瑞幸 App；必须清晰标注为本地 UI 演示，并在支付页停止。
- UI 加料过多会导致移动端拥挤。
- 决策逻辑复杂化后可能出现类型不一致或推荐解释对不上。

- 核对关键交互：

  - 手机端点击“执行点单”后进入模拟点单，并出现链路弹窗
  - 自动执行停在支付前，不创建真实订单、不扣券、不扣款
