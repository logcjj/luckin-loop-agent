## 执行方案

1. 先增强数据模型与业务内容：咖啡商品、营养/风味、门店、会员等级、券包、库存/排队、营销触达、Agent 配置。
2. 再增强决策逻辑：从单一规则推荐升级为多信号 scoring、券策略、门店策略、风险/隐私边界、fallback。
3. 优化 UI 信息架构：在首屏展示“业务闭环 + Agent 信心 + 推荐理由”，让对话、方案、画像、执行、复购都更充实。
4. 优化美术风格：更接近瑞幸蓝白、咖啡暖色辅助、App/小程序感，避免单一绿色主题。
5. 做移动端适配：rail/canvas 在窄屏重排为可横向选择或堆叠布局，保持按钮和文字不溢出。
6. 完善文档：说明架构、数据来源边界、真实系统接入方式、演示脚本和评估指标。
7. 验证：TypeScript build、页面截图/响应式检查、手动功能路径。

## 风险

- 直接抓取或声称使用真实瑞幸会员信息会有隐私和合规风险。
- 公开菜单、价格、优惠会随时间变化，必须避免写死为“官方实时”。
- UI 加料过多会导致移动端拥挤。
- 决策逻辑复杂化后可能出现类型不一致或推荐解释对不上。

## 验证方式

- `npm run build`
- 若项目提供测试则运行测试；如无测试，至少执行 TypeScript build。
- 启动本地 dev server 并检查桌面/手机宽度布局。
- 核对关键交互：
  - 切换会员
  - 切换场景
  - 输入 prompt
  - 确认/换兜底/模拟支付
  - 切换四个 canvas tab
